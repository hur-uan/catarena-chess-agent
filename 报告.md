# 基于 CATArena 的棋类 Agent 参数自学习系统报告

## 摘要

本项目实现了一个面向 CATArena 对战平台的棋类 Agent 参数自学习系统。系统以固定下棋程序为执行基础，根据对战反馈自动调整策略参数，并通过多层验证机制决定候选参数是否可以进入正式配置。

当前系统已经形成较清晰的实验闭环：Agent 先在 CATArena 环境中完成对局，然后系统读取对局日志、排名结果和历史记录，从中提取失败局面与性能反馈；随后生成候选策略参数，并通过局面预筛、自博弈评估、统计检验和历史回归判断候选是否可靠。只有候选策略在这些环节中都不弱于当前基线时，才允许被晋升为新的正式策略。

用较浅显的话说，本项目的 Agent 类似于“保持基本思考框架不变，在复盘后微调决策习惯；每次微调后先进行考试，考试合格才允许采用新参数”。这种设计显著提高了实验稳定性和结果可解释性。

## 1. 研究背景与问题定义

棋类 Agent 的能力提升通常依赖大量对局反馈。对于课程项目或正式实验而言，稳定性、可分析性和可复盘性是评价系统设计的重要标准。本项目因此采用受控的参数自学习路线：系统将下棋程序视为固定执行器，将可调整部分限制为策略参数 profile。每一轮实验中，程序根据反馈调整经过注册和约束的参数。

当前系统解决的问题可以概括为：

> 在固定棋类 Agent 运行时的前提下，如何利用平台对战反馈自动生成、验证并有条件接受更优的策略参数。

这种问题定义使系统既具有一定的自学习能力，又能保持实验过程的可复盘性。

## 2. 系统总体设计

当前系统采用 `profile-only` 参数自学习架构。其含义是：正式实验中，Agent 的 Python 源码保持固定，系统只调整策略参数文件中的受控参数。

当前正式实验的版本标识为：

- `optimization_mode`: `profile_only`
- `pipeline_version`: `profile_only_v2_internal_runtime`
- `runtime_policy`: `formal_internal_only`
- `formal_execution_backend`: `internal_engine`

这些标识会写入实验产物中，用于说明该轮实验属于当前正式管线。这样做的好处是可以避免不同历史版本的实验结果混在一起，保证结果分析时能够明确知道每个结果来自哪一种运行策略。

从系统结构看，当前项目可以分为两个主要部分：

| 部分 | 功能 |
| --- | --- |
| 运行时下棋模块 | 根据当前局面和策略参数选择合法走法 |
| 轮间优化模块 | 根据对战反馈生成和验证候选策略参数 |

二者的关系是：运行时模块负责“下棋”，轮间优化模块负责“总结经验并提出参数更新”。这种分工使系统能够在保持下棋接口稳定的同时，逐步改进策略表现。

## 3. 运行时下棋链路

当前正式运行时链路为：

```text
observation -> strategy_profile -> search_router -> internal engine -> legal guard -> move
```

其中，`observation` 是平台给出的当前棋局状态；`strategy_profile` 是当前使用的策略参数；`internal engine` 是系统内部搜索引擎；`legal guard` 用于确保最终输出走法合法。

这条链路的核心特点是固定和可控。正式实验中，Agent 默认只使用内部搜索引擎，不再默认调用外部 `black_numba` 引擎。这样可以保证被优化的参数确实作用于正式下棋过程。

通俗地说，如果系统优化的是内部引擎参数，那么正式比赛时也必须让内部引擎真正负责下棋。否则就会出现“训练的是一个系统，比赛用的是另一个系统”的问题，反馈信号也就失去意义。

因此，当前实验中的执行器和被调参数是一致的。这是本项目能够进行有效参数自学习的基础。

## 4. 策略参数 profile

本项目中的策略 profile 是 Agent 的可学习对象。它包含搜索深度、走法排序、估值权重、棋子价值等多类参数。当前正式在线优化并没有开放全部参数，而是只开放经过验证、风险较低的参数范围。

当前正式在线自治优化只开放一个参数块：

- `search.history_order`

当前真正允许在线调整的参数是：

- `search.tt_move_bonus`

该参数与搜索过程中的置换表走法排序有关。简单理解，它影响搜索引擎在考虑候选走法时，对历史搜索中较重要走法的优先级安排。

其他参数虽然在注册表中保留，但当前不进入正式在线优化。这样做的原因是参数面越宽，系统越难判断某次表现变化到底由哪个参数造成，也更容易产生不稳定候选。当前系统选择先在较小参数面上建立可靠闭环，再保证每次变化都可以被解释和验证。

## 5. 轮间自我提升流程

当前系统的自我提升流程可以表示为：

```text
CATArena 对战反馈
-> 失败类型分析
-> 失败局面集合构建
-> 候选参数生成
-> 自博弈与局面预筛
-> 统计检验
-> 历史回归
-> 晋升门控
```

这一流程体现了“先提出候选，再验证候选，最后决定是否采用”的思想。

具体来说，系统会读取 CATArena 产生的对战日志、排名文件和历史记忆记录，分析本轮是否出现非法走法、超时、崩溃或明显失利局面。随后，系统基于这些反馈构建失败局面集合，并尝试生成候选 profile。

候选 profile 不会直接上线，而是要和当前基线 profile 进行比较。比较方式包括局面预筛、自博弈评估和历史回归。只有当候选 profile 在这些验证中表现不弱于当前基线，且满足门控条件时，才允许写回正式策略配置。

用更直观的话说，系统每一轮都会问三个问题：

1. 这轮对局暴露了什么问题？
2. 是否能提出一个可能更好的参数版本？
3. 这个新版本是否经过验证，足够可靠到可以替换旧版本？

当前系统已经实现了这三个问题对应的基本流程。

## 6. 自博弈评估机制

当前系统已经具备本地成对自博弈评估机制。这里的“自博弈”不是指两个完全不同的程序互相对战，而是指两个不同策略 profile 在同一个固定引擎框架下进行对局比较。

在评估中，系统通常将候选 profile 作为一方，将当前基线 profile 作为另一方。为了减少先手优势的影响，同一开局会进行成对对局，即双方交换先后手后再比较综合结果。

这种设计有两个作用：

- 它可以判断候选参数是否真的优于当前参数。
- 它可以降低单局胜负偶然性对判断结果的影响。

通俗地说，候选参数不能只靠“碰巧赢了一盘”就上线，而是要在相对公平的对比条件下证明自己至少不比旧参数差。

当前自博弈评估记录的指标包括胜负情况、平均得分、走法数量、超时情况和崩溃情况等。这些指标不仅反映棋力表现，也反映运行稳定性。

## 7. 参数优化方法

当前系统采用 SPSA 思路进行参数搜索。SPSA 是一种适合高噪声环境的随机近似优化方法。棋类对战结果往往具有较强随机性和不稳定性，单次对局胜负并不能完全说明参数优劣，因此需要用较稳健的方式估计参数调整方向。

在本项目中，SPSA 的基本思想可以浅显理解为：

> 在当前参数附近生成两个轻微不同的版本，让它们进行比较，然后根据比较结果判断参数应该往哪个方向调整。

系统并不只依赖自博弈结果，还会结合固定局面上的预筛结果。这样可以避免候选参数只在少数对局中表现较好，却在关键局面上明显退化。

因此，当前优化信号由两部分构成：

- 局面预筛信号：候选在典型局面或失败局面上的表现。
- 对局比较信号：候选在本地成对自博弈中的表现。

这种组合使参数更新更加谨慎，也更适合当前受控实验环境。

## 8. 候选晋升门控

候选 profile 是否能成为正式 profile，由 promotion gate 决定。当前门控条件主要包括：

- 当前 Agent 接口验证通过。
- 候选 profile 相比基线确实发生有效变化。
- 调参策略认为候选达到可接受状态。
- 候选在反馈局面预筛中不弱于基线。
- 候选通过同一正式运行策略下的历史回归测试。

如果候选没有通过这些条件，系统会保存候选和评估报告，但不会将它写回正式配置。

这一机制是当前系统的重要安全设计。它保证了系统不是“每轮都强行进化”，而是“只有通过验证才允许进化”。在科研实验中，这一点非常重要，因为未经验证的自动更新可能会造成性能退化，并使实验结果难以解释。

## 9. 历史回归机制

当前系统在评估候选 profile 时，会参考历史 profile。为了保证比较公平，历史回归只选择与当前正式实验语义一致的历史记录，即满足：

- `runtime_policy == formal_internal_only`
- `formal_execution_backend == internal_engine`

这样可以避免把早期不同运行策略下的结果混入当前评估。

浅显理解就是：候选参数要和“同一考试规则下”的历史版本比较，而不是和使用不同引擎、不同规则或不同代码路线的旧版本比较。这样才能保证回归测试结果有意义。

## 10. 实验产物与可复盘性

当前系统每轮实验都会生成结构化产物，主要包括：

- `optimization_report.json`
- `round_record.json`
- `evaluation_record.json`
- `parameter_record.json`
- `strategy_profile_before.json`
- `strategy_profile_after.json`
- `candidate_profile.json`
- `feedback_fen_suite.json`

这些文件记录了从对局反馈到候选生成、评估和门控的完整过程。通过这些产物，可以复盘以下问题：

- 本轮对局结果如何？
- 是否发生非法走法、崩溃或超时？
- 系统提取了哪些关键失败局面？
- 是否生成候选 profile？
- 候选 profile 修改了哪些参数？
- 候选是否通过局面预筛？
- 候选是否通过历史回归？
- 候选是否最终晋升？

因此，当前系统不仅能运行，还能留下可分析、可追踪的实验记录。这对科研报告和实验复盘都很重要。

## 11. 当前正式样本说明

当前较适合作为报告依据的正式样本位于：

- `reports/formal_internal_mature_smoke/formal_internal_mature_002/optimization_report.json`
- `reports/formal_internal_mature_smoke/formal_internal_mature_002/evaluation_record.json`
- `reports/formal_internal_mature_smoke/formal_internal_mature_002/round_record.json`

该样本记录了当前正式管线的关键事实：

- 使用 `profile_only_v2_internal_runtime` 管线。
- 使用 `formal_internal_only` 运行策略。
- 正式执行后端为 `internal_engine`。
- 系统生成了候选 profile。
- 系统执行了反馈局面预筛。
- 系统执行了历史回归。
- 候选最终未通过 promotion gate，因此没有晋升。

候选未晋升并不表示流程失败。相反，它说明当前系统的门控机制发挥了作用：当候选没有在反馈局面上达到不弱于基线的要求时，系统会拒绝将其写入正式配置。

这可以作为报告中的一个重要结果：

> 当前系统已经能够完成候选策略生成、验证和拒绝流程，说明参数自学习闭环具备基本可运行性，同时具备防止低质量候选上线的保护机制。

## 12. 工程可靠性

当前仓库包含测试集合，覆盖 Agent 接口、合法走法、内部引擎、CATArena 集成、SPSA、SPRT、参数注册表、profile 优化、局面预筛和连续学习脚本等内容。

这些测试为当前 profile-only 管线提供了基本回归保障。对于本项目而言，测试的意义不仅是检查代码是否能运行，还包括验证以下性质：

- Agent 输出是否满足平台接口。
- 输出走法是否合法。
- 参数优化流程是否能生成结构化报告。
- 候选晋升逻辑是否遵守门控条件。
- 连续学习脚本是否能在多轮实验中保持产物结构一致。

因此，当前系统具备一定的工程稳定性基础，能够支持实验分析和课程报告撰写。

## 13. 系统特点总结

当前项目的主要特点可以概括为以下几点。

第一，系统采用固定运行时代码和可调策略参数分离的设计。这样既保留了自学习能力，又降低了运行时结构变化带来的不稳定性。

第二，系统使用内部引擎作为正式执行器，并保证当前可调参数直接作用于该执行器。这使优化目标和实际比赛行为保持一致。

第三，系统通过 CATArena 反馈、失败局面提取、自博弈评估、统计检验和历史回归构成闭环，使参数更新具有反馈依据。

第四，系统使用 promotion gate 控制候选上线，只有经过验证的候选才可能替换正式 profile。这使系统具备基本的安全更新机制。

第五，系统为每轮实验生成结构化产物，使实验过程可以被追踪、复盘和分析。

## 14. 结论

本项目当前已经实现了一个受控的棋类 Agent 参数自学习系统。系统以 CATArena 对战反馈为输入，以固定内部引擎为执行器，以策略 profile 为可调整对象，通过局面预筛、自博弈评估、统计检验、历史回归和晋升门控完成候选策略的生成与验证。

从科研实验角度看，该系统的价值在于将 Agent 的自我改进过程约束在可解释、可验证和可复盘的参数空间中。系统通过固定执行器和受控参数更新，在稳定性和自学习能力之间取得平衡。

当前最准确的结论是：

> 系统已经具备基于平台反馈的参数级自学习闭环，并能够通过多层验证机制阻止未达标候选策略进入正式配置。
